Q1
What is a branch in Git?

A branch is a lightweight, movable pointer to a specific commit in your repository's history. It lets you work on features, fixes, or experiments in isolation — without touching the main codebase.

git branch feature-x        # create a branch
git checkout feature-x      # switch to it
git checkout -b feature-x   # create AND switch (shorthand)

git branch                  # list all local branches
git merge feature-x         # merge into current branch
git branch -d feature-x     # delete it

Each branch is just a file containing a 40-character commit hash — nearly instantaneous to create. HEAD is a special pointer tracking which branch you're currently on.

main: A → B → C ↑ HEAD (if you're on main) feature-x: A → B → C → D → E ↑ HEAD (if you're on feature-x)
Q2
I have file1.c and file2.c in main, branch to "progress", work on file2.c — do I need to merge?

Yes. Branch off, do your work, then merge back. file1.c is untouched throughout — Git only tracks changes, not files wholesale.

git checkout -b progress    # branch off main
# ... edit file2.c ...
git add file2.c
git commit -m "updated file2"

git checkout main            # switch back
git merge progress           # bring changes in
git branch -d progress       # clean up

⚠️ If main also changed file2.c while you worked, you'll get a merge conflict and need to resolve it manually. Rare in solo work.

Q3
What if I only had file1.c and added code inside it?

Same concept — Git tracks changes within files, not just files appearing or disappearing.

// file1.c on main (original)
int main() {
    return 0;
}
// file1.c on progress (after edits)
void hello() {
    printf("hello\n");
}

int main() {
    hello();
    return 0;
}

Switching branches literally rewrites the file on disk to match that branch's snapshot:

git checkout main      # file1.c → old version (no hello)
git checkout progress  # file1.c → new version (with hello)
git checkout main
git merge progress     # main now has hello() too
